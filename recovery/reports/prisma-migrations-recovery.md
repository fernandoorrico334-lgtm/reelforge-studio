## Diagnosis

### What was recoverable

- The final `prisma/schema.prisma` was intact.
- Five late-stage migrations still had valid SQL:
  - `render_cinematic_v2_mode`
  - `audio_engine_v1`
  - `manual_intake_system`
  - `production_flow_channel_defaults`
  - `media_collector_v1`

### What was not safely recoverable as historical truth

These folders were missing `migration.sql` entirely:

- `init`
- `caption_templates_blueprint`
- `render_jobs_v1`
- `render_ops_v1`
- `research_collector_v1`
- `hybrid_visual_engine_v1`

Because the missing files span both the original schema foundation and multiple structural feature milestones, reconstructing the old incremental history would have required inventing historical SQL rather than restoring it faithfully.

## Why Rota B was chosen

Incremental reconstruction was rejected because:

- the original Git history is gone in this recovered workspace
- the missing migrations include foundational schema creation
- mixing a guessed early history with preserved late SQL would create high risk of drift or double-application

The safe recovery choice was:

1. preserve the entire old migration chain in `recovery/archived-migrations/pre-recovery-baseline/`
2. create one new recovery baseline from the final restored schema
3. validate that a fresh SQLite database can be created and managed by Prisma Migrate again

## Recovery steps performed

### Attempt 1: manual diff baseline

- A manual baseline was first generated from:
  - `prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script`
- That baseline was archived as:
  - `recovery/archived-migrations/manual-diff-baseline-attempt/`
- It was not kept as the active solution because Prisma Migrate in this Windows session still returned non-diagnostic schema-engine errors during `migrate deploy`.

### Important environment quirk discovered

- When `prisma/dev.db` did not exist, `prisma migrate status` and `prisma migrate deploy` in this session returned an unhelpful generic schema-engine error.
- After recreating an empty `prisma/dev.db`, the same commands started returning proper diagnostics.

### Final working solution

The active migration chain is now:

- `prisma/migrations/20260630015437_recovery_baseline/migration.sql`
- `prisma/migrations/migration_lock.toml`

This baseline was generated by Prisma itself using:

- `npx prisma migrate dev --name recovery_baseline --create-only --schema prisma/schema.prisma --skip-generate`
